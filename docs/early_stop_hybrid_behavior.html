Hybrid Early Stop Behavior (gsplat)

This document describes the implemented hybrid early-stop logic used by gsplat training. The design uses frequent trend monitoring but only allows hard stop decisions at eval steps.

Why Hybrid

  • Monitor trend between eval points so we do not miss slow plateaus.
  • Avoid false positives by confirming stop only on eval checkpoints.
  • Keep behavior stable across noisy minibatch loss updates.

Decision Flow

  1. Monitor phase every early_stop_monitor_interval steps: compute EMA loss and rolling improvement.
  2. If monitor improvement is too small, set early_stop_candidate=true.
  3. Eval confirmation phase at eval step: only then evaluate stop gates.
  4. If gates pass, write early-stop flag and end training gracefully after current step.
  5. Export still runs for early-stop runs (final + best artifacts are preserved).

Stop Gates (all required)

  • Minimum eval points reached.
  • Minimum completed step ratio reached.
  • Candidate state persisted at least one eval interval.
  • Eval-window relative improvement below threshold.
  • Eval-window volatility ratio below threshold.

Default Parameters

Parameter Default Description
auto_early_stop false Enable hybrid early-stop.
early_stop_monitor_interval 200 Fast monitor cadence in train steps.
early_stop_decision_points 10 Rolling window size for both monitor and eval decisions.
early_stop_min_eval_points 6 Minimum eval points before stop can be confirmed.
early_stop_min_step_ratio 0.25 Minimum fraction of max steps that must complete first.
early_stop_monitor_min_relative_improvement 0.0015 Monitor-window minimum improvement to avoid candidate state.
early_stop_eval_min_relative_improvement 0.003 Eval-window minimum improvement to avoid confirmed stop.
early_stop_max_volatility_ratio 0.01 Maximum normalized eval volatility allowed for stop.
early_stop_ema_alpha 0.1 EMA smoothing factor for monitor trend.

Recorded Outputs

  • metadata.json includes early_stop diagnostics and trigger reason.
  • Comparison summary includes stopped_early and early_stop_step.
  • Status updates expose candidate/trigger details for UI visibility.
Recommendation: keep early_stop_decision_points=10 for stable trend assessment. Smaller windows can react faster but are more sensitive to noise.
